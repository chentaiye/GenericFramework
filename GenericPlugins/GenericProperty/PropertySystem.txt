PropertySystem 需求说明

1. 项目目标
- 实现一个运行时 Property Detail Panel，输入单个或多个 UObject，即可在游戏运行时查看、搜索、分组、编辑、应用、重置属性。
- 目标是“功能心智接近 Unreal Editor Detail Panel 的运行时版本”，不要求 UI 逐像素复刻编辑器，但要求交互模型、层级结构、编辑行为、错误反馈尽量一致。
- 系统必须以反射自动生成属性树为主，不能依赖手工为每个业务对象单独写 ViewModel；同时保留手工扩展、自定义控件、自定义数据源的入口。
- UI 层必须采用 MVVM 解耦，View 不直接持有业务对象，不在 Widget 中写属性解析和业务逻辑。
- 系统必须覆盖“常见类型直接可编辑、复杂类型可展开编辑、未知类型可安全降级显示”的完整策略，任何不支持的情况都不能导致崩溃。

2. 核心原则
- 自动优先：默认通过 UPROPERTY 反射自动构建属性节点、分类、显示名、提示信息和编辑规则。
- 解耦优先：对象扫描、数据访问、状态同步、验证、展示控件、命令按钮必须分层。
- 安全优先：空对象、无效路径、对象销毁、GC、外部值变化、多人同步、未知类型都必须有明确处理策略。
- 扩展优先：允许按“属性类型 / 属性名 / 元数据 / GameplayTag / 路径”覆盖默认行为。
- 批量优先：支持 Apply、Reset、Dirty 标记、局部刷新、搜索过滤、多对象编辑。
- 兼容优先：兼容现有 `PropertyCollection`、`PropertyList`、`PropertyProxy`、`PropertyDataSourceCollection`、`PropertyViewModel`、`PropertyVisualData` 的模块结构，不推翻现有架构。

3. 典型使用场景
- 运行时调试面板：调 Actor、Component、Subsystem、DataAsset、配置对象。
- 玩家设置面板：图像、音频、输入、辅助功能等可持久化参数。
- 开发者工具：修改数值、触发命令、切换枚举状态、查看只读运行时信息。
- 关卡内对象检查器：选中一个场景对象后自动展示其可编辑属性。
- 批量编辑：同时选中多个对象，编辑共有属性。
- 配置编辑：编辑 SaveGame、Config UObject、DataAsset 的可保存字段。

4. 输入与输出
- 输入必须支持：
- 单个 UObject。
- 多个 UObject。
- Actor、ActorComponent、UserWidget、Subsystem、DataAsset、普通 UObject、Instanced UObject。
- 运行时动态切换目标对象。
- 只读模式、可写模式、自动应用模式、手动应用模式。
- 白名单、黑名单、分类过滤、搜索关键字、展开策略、显示高级属性开关。
- 输出必须包括：
- 面板级 ViewModel。
- 分类级 ViewModel。
- 属性节点级 ViewModel。
- 属性控件实例或警告控件实例。
- Dirty 状态、验证状态、可见性、可编辑性、混合值状态、错误信息。

5. 与现有模块的映射
- `UPropertyCollection` 作为面板根节点，负责多个分类列表的装配、Apply/Reset 总入口、搜索和全局状态。
- `UPropertyList` 作为分类或分组容器，负责一个分类下的属性列表展示。
- `UPropertyProxy` 作为当前检查会话或目标对象代理，负责属性集合生命周期、Dirty 汇总和统一刷新。
- `UPropertyDataSourceCollection` 保留为“属性来源桥接层”，新增自动反射生成版本，同时保留手工版本用于高度定制场景。
- `UPropertyViewModel` 作为所有属性节点的基类，统一承载显示信息、编辑状态、依赖关系、Dirty 状态和通知。
- `UPropertyVisualData` 负责“属性节点 -> Widget 类 / Option 类”的映射注册表。
- `UPropertySubsystem` 负责运行时代理注册、查找、切换、销毁和全局事件广播。

6. 反射扫描需求
- 默认只扫描可反射的 UPROPERTY。
- 必须支持递归扫描对象、结构体、容器中的子字段，形成树状 Property Node。
- 必须支持从 metadata 读取 DisplayName、ToolTip、Category、AdvancedDisplay、EditCondition、ClampMin、ClampMax、UIMin、UIMax、Delta、Units、MultiLine、PasswordField、AllowedClasses、DisallowedClasses、ExactClass、TitleProperty、ShowOnlyInnerProperties、InlineEditConditionToggle、NoResetToDefault 等常见规则。
- 必须支持按配置忽略指定属性：
- 属性名黑名单。
- 类型黑名单。
- 路径黑名单。
- metadata 标记黑名单。
- 自定义 Predicate。
- 必须支持按配置仅暴露指定属性：
- 属性名白名单。
- Category 白名单。
- metadata 白名单。
- 自定义 Predicate。
- 必须支持属性排序规则：
- Category 排序。
- Category 内属性排序。
- Advanced 属性折叠。
- 同名继承属性的覆盖规则。
- 必须支持在自动反射结果基础上做二次加工：
- 隐藏属性。
- 改名。
- 改 Tooltip。
- 替换控件。
- 替换编辑策略。
- 插入自定义 Action 节点。

7. 属性类型覆盖要求
- 所有被反射到的属性必须至少满足以下三种结果之一：
- 有专用编辑器，可直接编辑。
- 可展开为子字段进行编辑。
- 无法专用编辑时以只读文本或警告控件安全降级显示。
- 不允许出现“因类型未知直接丢失节点”或“点击即崩溃”的情况。

8. 必须直接支持的基础类型
- `bool`
- `uint8 / int32 / int64 / float / double` 及其它常见数值类型
- `enum` 和 `bitmask`
- `FString`
- `FName`
- `FText`
- `FGameplayTag`
- `FGameplayTagContainer`
- `TSubclassOf<>`
- `TObjectPtr<UObject>`、裸 UObject 引用、软引用、弱引用、软类引用
- 常见路径类：`FSoftObjectPath`、`FSoftClassPath`

9. 必须直接支持的常见结构体
- `FVector2D`
- `FVector`
- `FVector4`
- `FRotator`
- `FQuat`
- `FTransform`
- `FColor`
- `FLinearColor`
- `FIntPoint`
- `FIntVector`
- `FMargin`
- `FDateTime`
- `FTimespan`
- `FGuid`
- 自定义 UStruct：默认以“可展开子字段”方式支持；若子字段也无法编辑，则降级为只读说明。

10. 必须支持的容器类型
- `TArray`
- `TSet`
- `TMap`
- 容器元素类型至少支持所有“基础类型”和“常见结构体”。
- 容器必须支持展开查看元素。
- `TArray` 必须支持增删、清空、复制元素、移动顺序。
- `TSet` 必须支持增删和唯一性校验。
- `TMap` 必须支持增删键值对、键唯一校验、键和值分开编辑。
- 对于容器中元素为 UObject / UStruct 的情况，必须支持递归展开。
- 当容器元素类型无法安全编辑时，容器本身也必须有只读降级显示。

11. 对象引用与子对象要求
- UObject 引用属性必须支持：
- 显示当前对象名、类名、路径。
- 清空引用。
- 重新选择引用。
- 根据 `AllowedClasses / DisallowedClasses / ExactClass` 过滤候选对象。
- 区分世界内对象引用与资源引用。
- 对 `Instanced` / `EditInlineNew` / `DefaultToInstanced` 类型，必须支持在权限允许时新建、替换、删除子对象。
- 对空引用但可实例化的对象，必须支持“创建实例”入口。
- 对引用失效、对象 PendingKill、资源未加载等情况，必须显示错误状态，不允许崩溃。

12. 多对象编辑需求
- 输入多个对象时，系统必须计算“可共同编辑属性交集”。
- 多对象编辑必须支持：
- 相同类对象。
- 继承链兼容对象。
- 混合值状态显示。
- 仅对交集属性开放编辑。
- 值不一致时显示 Mixed Value。
- 用户输入新值后，必须一次性写回所有合法目标对象。
- 当容器结构不一致、Map Key 不一致、对象类型不一致时，必须进入只读或受限编辑模式，并提示原因。

13. 编辑行为需求
- 每个属性必须维护：
- Source Value：当前数据源值。
- Current Value：当前编辑值。
- Default Value：类默认值或自定义默认值。
- Dirty：当前值是否相对 Source Value 已变更。
- Invalid：当前输入是否合法。
- 手动应用模式：
- 用户编辑只更新 Current Value，不立即写回对象。
- 支持单属性 Apply、分类 Apply、面板 Apply。
- 支持单属性 Reset、分类 Reset、面板 Reset。
- 自动应用模式：
- 用户编辑后在验证通过时自动写回。
- 需要防抖或提交时机策略，避免每次按键都重建整棵树。
- Reset 必须至少支持两种语义：
- 重置到面板打开时 / 最近一次 Apply 后的 Source Value。
- 重置到类默认值或业务默认值。
- 需要可配置选择默认 Reset 语义。

14. 验证与错误反馈
- 所有输入型属性都必须先过验证再提交。
- 验证至少包括：
- 类型解析是否合法。
- 数值范围是否合法。
- Enum 是否属于可选项。
- GameplayTag 是否有效。
- 对象引用是否满足类过滤。
- Map Key / Set Element 是否唯一。
- 自定义业务验证器是否通过。
- 验证失败时必须：
- 保留用户输入上下文。
- 不写回目标对象。
- 在属性级显示错误文本或警告图标。
- 将属性标记为 Invalid。
- 面板级可汇总错误数量。

15. 外部变更同步
- 当目标对象属性在面板外部被修改时，系统必须支持刷新同步。
- 同步策略至少支持：
- 立即刷新。
- 延迟刷新。
- 手动刷新。
- 当外部值变化发生在本地 Dirty 未提交期间，必须有冲突策略：
- 以本地编辑为准。
- 以外部新值为准。
- 提示用户冲突。
- 同步时必须避免无限通知循环、重复 Apply、递归刷新。

16. 可见性、可编辑性与依赖关系
- 必须支持属性级别的 IsVisible、IsEditable、IsEnabled、IsAdvanced。
- 必须支持基于其它属性值动态控制显示和可编辑状态。
- 必须支持 `EditCondition` 与自定义依赖关系。
- 属性依赖变化后，依赖项必须自动刷新。
- 必须检测并阻止依赖死循环、递归刷新风暴。
- 当依赖失效时，属性应进入禁用或隐藏状态，而不是保留错误可编辑态。

17. UI 与交互需求
- 界面必须支持：
- 分类标题。
- 搜索框。
- 高级属性显示开关。
- 展开 / 折叠。
- Tooltip。
- Dirty 标识。
- Invalid 标识。
- Reset 按钮。
- 只读显示。
- 混合值显示。
- 属性控件必须由注册表按“类型 / 名称 / metadata / tag”匹配生成。
- 若控件不支持当前 ViewModel，必须显示 Warning Widget，而不是静默失败。
- 搜索必须支持：
- 按显示名搜索。
- 按属性名搜索。
- 按 Category 搜索。
- 按路径搜索。
- 按 Tag 搜索。
- 面板在属性很多时必须支持列表虚拟化或懒加载，避免一次性创建过多 Widget。

18. MVVM 解耦要求
- View 不允许直接读写目标 UObject。
- View 只能绑定 ViewModel 字段、命令和事件。
- ViewModel 不直接依赖具体 Widget 类。
- 反射扫描层不依赖具体 UMG 实现。
- 数据写回必须经过统一 DataSource / PropertyHandle 层。
- 命令按钮、Action 项也必须走 ViewModel / Command 机制，不允许在按钮点击里直接写业务对象。

19. 数据访问层要求
- 需要统一的 Property Handle / DataSource 抽象，屏蔽反射路径读写细节。
- 路径必须支持：
- 直接属性。
- 嵌套 Struct 字段。
- 数组元素。
- Set / Map 元素。
- 子对象字段。
- 必须能报告路径是否可读、可写、可展开、是否需要实例化中间对象。
- 失败时要能返回错误原因，不允许只返回空值。
- 需要支持自动反射数据源和手工自定义数据源两种模式共存。

20. 生命周期与安全性
- 面板创建、目标切换、对象销毁、Widget 销毁、场景切换时都必须正确解绑。
- 被检查对象被 GC、销毁、卸载、切关卡后，界面必须进入安全状态。
- 不允许保留悬空引用。
- 切换目标对象时必须先清理旧会话，再建立新会话。
- 多个面板同时查看同一对象时，必须有明确的同步策略。

21. 保存与网络语义
- 系统本身只负责“编辑运行时对象状态”，不默认承担持久化。
- 但系统必须提供保存扩展点，支持：
- SaveGame。
- Config。
- DataAsset。
- 自定义序列化。
- 对复制属性必须明确约束：
- 非 Authority 默认只读，或通过显式命令路由到服务器。
- 不允许客户端静默篡改应由服务器控制的复制状态。
- 对调用 `OnRep`、广播事件、事务录制等副作用要有统一策略，避免重复触发。

22. 权限与暴露控制
- 必须支持按对象、类型、属性路径配置为只读或隐藏。
- 必须支持调试字段与玩家可见字段的权限隔离。
- 对敏感字段如密码、Token、内部调试对象引用，必须支持脱敏或完全隐藏。
- 发布版本应支持关闭部分调试面板或限制到开发者权限。

23. 性能要求
- 反射结果应按 Class 缓存，避免重复全量解析。
- 属性值刷新应尽可能增量化，避免任何单属性变更都重建整个面板。
- 搜索过滤应仅更新可见节点，不重复创建全部节点。
- 大容器、大对象树必须支持惰性展开。
- 运行时不允许因打开一个大对象详情面板造成明显卡顿或持续 Tick 开销异常。

24. 本地化与可用性
- DisplayName、Tooltip、分类名、错误文本必须支持本地化。
- 键盘、鼠标、手柄导航应可用。
- 数值输入、枚举选择、对象选择、颜色选择等常见交互要符合用户直觉。
- 需要支持只读态视觉区分、脏状态视觉区分、错误态视觉区分。

25. 扩展点需求
- 允许为某个类型注册专用 ViewModel。
- 允许为某个属性路径注册专用 Widget。
- 允许为某个 metadata / tag 注册专用 Option。
- 允许注册自定义 Validator、Formatter、Parser、Serializer。
- 允许注册自定义 Command / Action 节点。
- 允许在自动生成结果前后插入加工步骤。
- 允许业务方完全替换某个分类或某个属性的默认生成逻辑。

26. 与现有实现兼容的落地建议
- 保留现有 `UPropertyDataSourceCollection` 手工生成模式，用于高度定制面板。
- 新增“自动反射 DataSourceCollection / PropertyGenerator”，用于“给一个 UObject 自动出面板”的主流程。
- 保留 `UPropertyVisualData` 做视觉映射，但需要把映射规则扩展到“类型、名称、元数据、路径、Tag”多个维度。
- 继续使用 `UPropertyProxy` 汇总 Dirty 与 Apply/Reset，但其职责需要从“手工属性集合代理”扩展为“运行时检查会话代理”。

27. 非功能性硬约束
- 任何非法输入、空对象、路径失效、类型不匹配都不能崩溃。
- 任意一个属性控件加载失败时，整面板仍需继续工作。
- 日志必须能定位到对象、属性路径、失败原因。
- 不允许把编辑器专用 API 当成运行时强依赖。

28. 最低验收标准
- 给一个普通 UObject，能自动生成分类和属性列表。
- 常见基础类型可正常读写，Apply/Reset 生效。
- Struct 可展开编辑，子字段变更能正确回写。
- Array/Set/Map 至少能安全显示并对支持的元素类型进行编辑。
- 对象引用能显示、选择、清空，并遵守类过滤。
- 多对象编辑能识别交集属性和 Mixed Value。
- 搜索、分类、Dirty、Invalid、只读状态都能正确表现。
- 外部改值后面板可刷新，不产生死循环。
- 不支持的属性类型显示 Warning，不崩溃。

29. 分阶段实现建议
- 第一阶段：单对象自动反射、基础类型、常见结构体、Apply/Reset、搜索、Dirty、Warning 降级。
- 第二阶段：对象引用、容器编辑、多对象编辑、EditCondition、外部变更同步、权限控制。
- 第三阶段：复杂自定义 Struct、Instanced 子对象、网络权限、保存策略、性能优化、完整扩展注册表。

30. 最终交付定义
- 最终系统不应是“示例性质的几个手工 ViewModel 控件集合”，而应是一个可给任意合法 UObject 挂载并自动生成运行时详情面板的通用框架。
- 该框架必须在“自动生成能力、扩展能力、稳定性、性能、交互一致性”五个维度达到可在项目内广泛复用的标准。
